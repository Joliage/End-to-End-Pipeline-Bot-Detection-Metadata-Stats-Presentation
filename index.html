Stage 1 — Raw Data

SOLR JSON Input Structure

05 / 32

One SOLR record

{ "ip": "2400:1a00:b080:6caa:dd3:...", "userAgent": "Mozilla/5.0 ...", "isBot": false, "id": "c814c4e0-f4e5-4873-...", "owningItem": ["5080228a-2270-..."], "owningColl": ["e5a652c3-655e-..."], "owningComm": ["832fd03d-8fa8-..."], "bundleName": ["ORIGINAL"], "statistics_type": "view", "time": "2025-04-30T18:22:11.514Z", ... }

Why we don't trust isBot

  • The isBot field comes from SOLR's own heuristics — known to be inaccurate, especially for residential proxies and rotating VPNs
  • We re-classify every IP independently using IPQS for ground truth
  • The fields we actually rely on:
    • ip — for bot detection lookup
    • owningItem — links the event to a publication
    • bundleName — distinguishes VIEW vs DOWNLOAD
    • time — for daily/monthly aggregation

Stage 2 — Bot Detection

Smart IPQS Integration

06 / 32

Cache-aware lookup

  • Step 1. Extract unique IPs from raw_data using a Python set
  • Step 2. Query bot_mapping.distinct("ip") to find IPs already classified
  • Step 3. Compute the difference — only the new IPs need an API call
  • Step 4. For each new IP, call IPQS, classify, and upsert into bot_mapping

Why this matters

  • IPQS free tier: 5,000 lookups. Naively re-checking all IPs would burn the quota in days
  • The cache makes the system idempotent — safe to re-run
  • For our 1,000-record sample: 369 unique IPs → 369 API calls on first run, 0 on re-runs
  • 7 security flags checked: is_vpn, is_proxy, is_tor, is_hosting, is_relay, is_mobile, is_abuse — any flag = bot

Stage 3 — Filter

Building filtered_ips Collection

07 / 32

Filtering.py logic

  • Load all raw events from JSON
  • Pre-build a fast in-memory dict: {ip → is_bot}
  • For each event, check the dict — drop if bot
  • Insert clean events into filtered_ips
  • Default: missing IP or unknown IP = treat as bot (safe default)

Stage 4 — Enrich

UNESCAP Repository API

08 / 32

The problem

Raw events only contain a publication ID (owningItem) — no title, no author, no program of work. We need to enrich each event with full metadata before we can present meaningful stats.

Solution: use the public DSpace API at repository.unescap.org/server/api/core/items/{id} to fetch metadata per publication.

What we extract

  • publication — id, title, handle, dateIssued, type
  • programs — programOfWork, areasOfWork (multiple per pub)
  • publisher — publisher, publisherPlace, rights
  • Three separate MongoDB collections to keep schemas normalized
  • Uses cloudscraper to bypass DSpace's bot challenge headers

Stage 4 details

Sample Repository API Response

09 / 32

GET /server/api/core/items/{id}

{ "id": "5080228a-2270-...", "handle": "20.500.12870/6128", "metadata": { "dc.title": [{"value": "Asia-Pacific Sustainable Development Report 2025"}], "dc.date.issued": [{"value": "2024-09-15"}], "dc.type": [{"value": "Report"}], "dc.publisher": [{"value": "United Nations ESCAP"}], "escap.programmeOfWork": [ {"value": "Environment and development"} ], "escap.areasOfWork": [ {"value": "Climate change and natural resources"} ] } }

Why three collections?

  • publication has a 1:1 relationship with a publication ID — core fields kept together
  • programs can have multiple entries per publication (programmeOfWork + areas)
  • publisher is conceptually distinct — it tells you who issued the work, not what the work is
  • Normalizing into separate collections allows independent queries and easier joins later

Stage 5 — Categorize

VIEW vs DOWNLOAD

10 / 32

The DSpace 7 SOLR rule

  • SOLR statistics records have a bundleName field (an array of strings)
  • If bundleName contains "ORIGINAL" → it was a file download (PDF, document)
  • Otherwise (THUMBNAIL, empty, etc.) → it was a page view
  • This rule is documented in the DSpace 7 docs and confirmed by the community forum

Distribution in our sample

bundleName distribution (1000 records): ORIGINAL → 638 events (DOWNLOAD) THUMBNAIL → 256 events (VIEW) (empty) → 106 events (VIEW) After bot filtering (608 records): ~388 downloads ~220 views

Reports get downloaded more often than viewed in-browser — typical for a research repository where users save PDFs offline.

Stage 6 — Aggregate

Daily & Monthly Stats

11 / 32

publication_stats (daily)

  • Schema: {publication_id, date, views, downloads}
  • Built once after stages 3-5, then refreshed when new data arrives
  • One row per publication per day

monthly_stats

  • Schema: {publication_id, year, month, views, downloads}
  • Computed by rolling up publication_stats
  • This is what the web UI displays in the search result

The MongoDB pipeline

  • $unwind on owningItem — flatten the array
  • $addFields — convert ISO time to Date, extract year/month
  • $group by (publication_id, year, month)
  • $sum with $cond — count views and downloads in one pass
  • upsert into monthly_stats — idempotent

Two layers, two purposes

  • Standalone scripts — bot_mapping, Filtering, publication_ingest. Run once (or on a schedule) to prep data. They populate MongoDB.
  • Flask app (main.py) — the live web service. On startup it runs the aggregation pipeline (idempotent), then serves search requests through the routes.
  • MVC discipline — every controller depends on a model, never directly on PyMongo. This makes the code testable and replaceable.

What gets stored

  • ip — the IP address (key)
  • is_bot — boolean computed from the 7 IPQS flags
  • response — full IPQS API payload (audit trail)

Why store the full response?

  • If we change the bot decision logic later (e.g. weight is_relay differently), we can re-classify without re-querying the API
  • The cached response includes geolocation and ASN — useful for future analytics
  • Free tier: 5,000 lookups. The cache means we never spend a lookup twice

Research Notes

How DSpace 7 Categorizes Events

24 / 32

Sources investigated

  • DSpace 7 official docs — Statistics & Reporting section
  • DSpace community forum (groups.google.com) — many threads on bundleName meaning
  • DSpace GitHub — source of SolrLoggerServiceImpl
  • The actual SOLR data — empirical analysis of the bundleName values present

What the data confirms

  • statistics_type alone is not enough — almost every record is "view" (995/1000 in our sample)
  • The real distinguisher is bundleName:
    • ORIGINAL — bitstream download (PDF, document, dataset)
    • THUMBNAIL — thumbnail image fetch (counts as a view)
    • (empty) — landing page view (counts as a view)
  • Other possible bundles (LICENSE, TEXT, BRANDED_PREVIEW) are rare in the public-facing repo

Robustness

Error Handling & Rate Limiting

25 / 32

API failures

If the IPQS call times out or returns an error, the IP is recorded as is_bot=True (safe default) with the exception stored in the response field for later debugging.

Rate limiting

1.0-second sleep between IPQS API calls — measured against elapsed time so we don't accidentally sleep too long. Keeps us under the free-tier rate limit comfortably.

Cloudscraper for repo

The UNESCAP repository sits behind a Cloudflare bot challenge. cloudscraper emulates a Chrome browser to pass the JS challenge and retrieve metadata reliably.

What Takes Time, What Doesn't

The slow stages are one-time setup. Once bot_mapping and the publication metadata are populated, the application serves search requests instantly. Re-running the scripts is cheap because the cache + skip-existing logic skip everything already done.
